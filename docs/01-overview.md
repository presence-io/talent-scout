# 01: 目标与原则

## 1. 项目愿景

构建一个可持续迭代的人才发现引擎（而不是一次性脚本），自动从公开数据中发现 AI Coding 时代优秀的中国开发者，输出可供招聘团队直接行动的候选人清单。

## 2. 目标人群画像

"AI Coding 时代优秀中国开发者"的特征，按优先级从高到低：

### P0 — 必须是优秀开发者

这是最内核的需求。无论在哪个时代，优秀的开发者都是稀缺的。衡量标准包括但不限于：

- 在开源项目中有持续、高质量的技术产出
- 拥有可观察的技术影响力（项目被采用、被引用、被协作）
- 具备从零到一的工程建设能力，而不只是参与维护

AI 能力是一个很大的加分项，但不能替代工程素养本身。

### P1 — 对 AI Coding 生态有实质贡献

不止是 AI 工具的使用者，而是对生态有正向输出。例如：

- 构建了被他人使用的 MCP server、Agent 工具、AI 工作流
- 开源了高质量的 AI 辅助开发配置或最佳实践
- 在多个项目中系统性地推广 AI Native 工作方式

贡献的影响力和被采纳程度决定了这个维度的得分。

### P2 — Coding 生产效率显著高于基准

开发者是否使用了 AI 工具，从外部不一定能直接看出来。但如果生产效率显著高于同语言、同领域的一般水平，则间接说明他们已经有效驾驭了 AI 工具。

衡量需要注意：

- 分语言设置基准参数（不同语言的代码密度差异很大）
- 除代码量外，代码质量也需要衡量（通过仓库的关键文件分析推断）
- 在积累足够样本后，基准参数应自动回归更新

### P3 — 善于使用 AI 工具

CLAUDE.md、.cursorrules、Co-Authored-By 等公开信号能证明开发者对 AI 工具有主动探索。这说明持续学习意愿和工具适应力，但单独不足以判断一个人是否"优秀"。

## 3. 必须警惕的反模式

### 不要把"用了 AI 技术"等同于"优秀"

AI Coding 时代的开发者，并不一定直接使用 AI 相关技术栈（LLM、LangChain、RAG 等）。不应对使用 AI 技术的项目赋予额外权重。

更需要警惕的是：近一年来大量开发者实验性地制作了各种 AI 玩具项目。这些项目：

- 无法证明开发者是否优秀
- 可能反而说明开发者相对初级（跟风而非深耕）
- 在评估中应被降权或忽略

**区分标准**：项目是否被他人真正采用和持续使用，而不是只看技术栈是否"AI"。

### 不要用信号密度替代候选人质量

v1 的核心错误是：把"有更多 AI 工具表面信号"的人排在前面，而不是把"真正优秀且值得联系的中国开发者"排在前面。新设计必须以候选人质量为排序目标。

## 4. 核心设计原则

### 原则 1：中国开发者身份是硬过滤条件

不是一个得分项，而是进入候选池的前置条件。先确认身份，再评估能力。

### 原则 2：多轴评估，拒绝单一总分

不用一个叠加分数决定一切。技术实力、AI 深度、可联系性、岗位匹配度分开建模，最终排序可组合但各轴独立可审计。

### 原则 3：所有结论必须保留证据

每个判断（"是中国人"、"技术 8 分"、"建议联系"）都必须可回溯到具体证据，而不是黑盒打分。

### 原则 4：确定性逻辑先行，AI 只处理模糊问题

规则能解决的不用 AI（去重、计数、pattern 匹配、API 编排）。AI 只用于模糊判断（身份推断、内容理解、候选人摘要）。

### 原则 5：系统具备自我进化能力

项目保留内置的 skill 基线来沉淀经验，但运行时新获得的知识不直接修改任何 npm 包内容。新的经验应以兼容 patch 的形式写入工作区，在后续执行时叠加到内置技能之上。

### 原则 6：公开分发是第一等约束

项目最终不是仅在本地 monorepo 内运行，还要以两种形态对外分发：

- `@talent-scout/*` workspace 包作为公开 npm package 发布，承载绝大部分业务逻辑
- `@talent-scout/skills` 作为唯一对外的 OpenClaw / ClawHub skill 包发布，负责声明 `SKILL.md`、整合命令面，并调用其他 npm 包暴露的 CLI 或查询接口

因此包边界、CLI 边界、查询接口、版本策略、发布流水线都必须从设计阶段明确，而不是在项目后期再补。

## 5. 与 v1 的关键差异

| 维度 | v1 | v2 |
|------|----|----|
| 中心假设 | 先抓 AI 信号，再判断是不是中国人 | 先构建中国开发者池，再评估 AI 深度和技术水平 |
| 技术栈 | Python 脚本 | Node.js (TypeScript) monorepo |
| AI 调用 | 未实现 / 未来直接调 API | 通过 OpenClaw agent 间接调用 |
| 数据存储 | 扁平 JSON 文件 | 分层 JSON (raw / processed / evaluated) + 缓存 |
| 排序目标 | AI 信号权重总分 | 多轴独立评估 + 组合排序 |
| 身份识别 | location 关键词（未实际接入） | 多信号 + AI 推断，输出 confidence |
| 可视化 | 无 | Astro + TailwindCSS + DaisyUI Dashboard |
| 持续运维 | 手动运行 | OpenClaw cron 定期调度 |
| 自我进化 | 无 | SKILLS.md + OpenClaw memory |
